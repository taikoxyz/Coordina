# Research: Multi-Agent Orchestration Patterns

> Status: Initial research — March 2026
> Covers: AutoGen, CrewAI, LangGraph, OpenAI Swarm; Lead/orchestrator patterns

---

## Overview

Multi-agent orchestration is a fast-evolving space. The core problem: how do multiple AI agents coordinate on tasks without creating loops, confusion, or conflicting actions? Different frameworks answer this differently.

For our product, OpenClaw handles the actual agent runtime and inter-agent communication. But understanding orchestration patterns informs how we design the Lead Agent concept and the UI for team structure.

---

## AutoGen (Microsoft)

**Philosophy**: Multi-agent conversation as the core abstraction. Agents coordinate by talking to each other.

**Orchestrator pattern**:
- A designated orchestrator agent receives tasks, distributes to specialized workers across layers, aggregates results
- Communication via `send_message()` API — flexible, non-sequential
- Supports both centralized (simpler control, potential bottleneck) and decentralized (distributed, more resilient) topologies

**Key strength**: Natural fit for LLMs — dialogue is how they naturally operate.

**Best for**: Research-oriented teams, exploratory tasks, flexible workflows.

---

## CrewAI

**Philosophy**: Role-based orchestration. Think in terms of team roles and responsibilities.

**Core abstractions**:
- **Flows**: Event-driven workflows managing state and execution order
- **Crews**: Groups of autonomous agents collaborating on delegated tasks within a Flow

**Key strength**: Intuitive for product teams — maps directly to human organizational structures. Fast to get started.

**Best for**: Product development teams, marketing teams, structured business workflows.

---

## LangGraph

**Philosophy**: Graph-centric orchestration. Maximum control and compliance.

**Core concept**: Directed acyclic graphs (DAGs) where:
- Nodes = agent actions or operations
- Edges = control flow and data flow between nodes

**Key strength**: Production-grade state management, observability, and granular control. Enterprise-ready.

**Best for**: Mission-critical systems, compliance-heavy domains, teams that need full auditability.

---

## OpenAI Swarm (Educational)

> Note: Swarm has been superseded by OpenAI Agents SDK for production use. Swarm is educational.

**Philosophy**: Lightweight, stateless agents with explicit handoffs.

**Key patterns**:
- **Agents and Handoffs**: Agents explicitly hand off conversations to other agents; no opaque orchestration
- **Routines**: Natural language instructions (system prompts) + associated tools define what an agent does

**Key strength**: Clarity and observability. You always know which agent is active.

**Relevance**: The "handoff" pattern is a useful mental model even if we don't use Swarm directly.

## Lead Agent Pattern — Synthesis

Across all frameworks, the "lead" or "orchestrator" agent has consistent characteristics:

1. **Single entry point**: All external communication enters through the lead. Other agents are internal.
2. **Task decomposition**: The lead breaks complex requests into subtasks for specialized agents.
3. **Result aggregation**: The lead collects results and synthesizes a coherent response.
4. **State awareness**: The lead maintains awareness of the overall team's state and progress.
5. **Escalation handling**: When a subtask fails, the lead decides whether to retry, reassign, or surface the issue to the human.

### Implications for our product

- The Lead Agent is always agent index 0 in a team
- The Mac app routes all human-initiated interactions to the lead by default
- The Mac app exposes the lead's gateway API as the "team endpoint"
- Admins can bypass the lead and talk directly to any agent for debugging/observation purposes
- The lead's SOUL.md should include orchestration instructions (generated by the app based on team structure)

## Human-in-the-Loop Patterns

Multi-agent systems benefit from well-defined human escalation points:

| Pattern | Description | When to use |
|---------|-------------|-------------|
| **Approval gate** | Agent pauses and asks human before proceeding | High-stakes actions (deploy, delete, send email) |
| **Observation mode** | Human can watch agent conversations in real-time | Debugging, onboarding |
| **Direct injection** | Human talks directly to any agent | Bypassing lead for specific instructions |
| **Memory inspection** | Human reads agent memory files | Understanding agent context/state |

Our product supports all four:
- Approval gates: configured per-agent in skills/tools
- Observation mode: future feature (v2)
- Direct injection: admin can select any agent to chat with
- Memory inspection: read-only MEMORY.md view in agent detail panel

## Agent Topology Considerations

### Star topology (recommended for v1)
```
     Human
       │
    Lead Agent
   /     |     \
Agent A  B    Agent C
```
- Simple, predictable routing
- Single exposed endpoint (lead's gateway)
- Easy to reason about for admins

### Mesh topology (future)
```
Human ─── Lead ─── Agent A ─── Agent B
              │                    │
           Agent C ────────────────┘
```
- Agents can communicate peer-to-peer
- More resilient but harder to observe
- OpenClaw supports this via its gateway routing

**v1 recommendation**: Star topology. Lead is the orchestrator; all agent-to-agent communication goes through it or is handled by OpenClaw's internal routing.
