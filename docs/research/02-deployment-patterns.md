## GKE for Stateful AI Agents

### Why StatefulSets (not Deployments)

Each OpenClaw agent has its own persistent workspace (config files, memory, skills). This requires:
- **Stable pod identity** — pod name stays consistent across restarts
- **Dedicated persistent storage** — each replica gets its own unique PersistentVolumeClaim

**Deployments** should only use PVCs with `ReadWriteMany` mode. For per-agent isolated storage, **StatefulSets** are the correct primitive.

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: agent-alice
spec:
  selector:
    matchLabels:
      app: agent-alice
  serviceName: agent-alice
  replicas: 1
  template:
    spec:
      containers:
        - name: openclaw
          image: openclaw/openclaw:latest
          ports:
            - containerPort: 18789   # OpenClaw gateway
          volumeMounts:
            - name: workspace
              mountPath: /root/.openclaw
  volumeClaimTemplates:
    - metadata:
        name: workspace
      spec:
        accessModes: ["ReadWriteOnce"]
        resources:
          requests:
            storage: 10Gi
```

### Persistent Volume Best Practices

- **Zone affinity**: Set `WaitForFirstConsumer` binding mode on StorageClass — provisions disks in the same zone as pods
- **High availability**: Regional persistent disks replicate across two zones in the same region, enabling failover during zonal outages
- **Storage class**: Use GKE's standard provisioner; disks automatically match pod scheduling constraints

## Exposing the API Server Pod

The OpenClaw gateway (port 18789) on the lead agent pod is the Mac app's entry point to the team. This must be exposed outside the cluster securely.

### GKE Ingress Architecture

```
Internet
  │
  ▼
Google Cloud Load Balancer (GCLB)
  │ (via GKE Ingress controller)
  ▼
Service (ClusterIP) → lead-agent pod :18789
```

GKE Ingress controller:
- Automatically creates Google Cloud Load Balancers
- Uses **Network Endpoint Groups (NEGs)** for container-native load balancing (default in modern GKE)
- Directly targets pods instead of nodes — lower latency, better health checks
- Automatically creates firewall rules for Google's health check IP ranges

### TLS / Auth

- TLS termination at the load balancer via GKE-managed certificates or cert-manager
- OpenClaw gateway auth token passed in headers — the Mac app stores this token per deployment
- Option: wrap behind **Google Cloud Identity-Aware Proxy (IAP)** for Google account-level auth

### Service definition

```yaml
apiVersion: v1
kind: Service
metadata:
  name: lead-agent-svc
  annotations:
    cloud.google.com/neg: '{"ingress": true}'  # container-native LB
spec:
  selector:
    app: agent-alice  # lead agent
  ports:
    - port: 80
      targetPort: 18789
  type: ClusterIP
---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: team-ingress
spec:
  rules:
    - host: team-slug.yourdomain.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: lead-agent-svc
                port:
                  number: 80
```

## Helm + Terraform Split

### Terraform: Cluster provisioning

Terraform manages the GKE cluster itself:
- VPC, subnets, firewall rules
- GKE cluster and node pools
- IAM service accounts
- DNS records

Tracked in the team spec repo under `infra/terraform/`.

### Helm: In-cluster application deployment

Helm manages what runs inside the cluster:
- One Helm release per team
- Chart values generated by the Mac app from the team spec
- Agent pods, services, ingress, network policies

Tracked in the team spec repo under `deploy/helm/`.

### Why split them

- Terraform is the right tool for cloud resource provisioning (idempotent, state-tracked)
- Helm is the right tool for K8s application lifecycle (upgrade, rollback, templating)
- The OpenClaw K8s operator ships as a Helm chart — use it directly

## GKE Authentication from Mac App

Two candidate approaches (decision deferred):

### Option A: Google OAuth + Workload Identity

- User signs in with Google account via OAuth
- App requests GKE API scope
- Workload Identity maps Google identity to K8s RBAC
- Pros: No credentials to store, uses existing Google account
- Cons: More complex OAuth setup, requires Google Cloud project config

### Option B: Service Account JSON Key

- User creates GCP service account with GKE permissions
- Downloads JSON key, uploads to Mac app
- App stores key in local keychain / encrypted store
- Pros: Simple, well-understood pattern
- Cons: Long-lived credential risk, manual rotation

**Recommendation**: Support both; default onboarding uses Google OAuth (safer), power users can use service account for automation.
